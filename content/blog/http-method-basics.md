## HTTPメソッドとは

HTTPメソッドとは**クライアントがリソースに対して行いたい処理をサーバーに伝えるもの**です。  
URIが「何を」を示すのに対し、メソッドは「どうする」を示します。

各メソッドとその用途は以下のとおりです。

| メソッド  | 用途                                                     |
| :-------- | :------------------------------------------------------- |
| GET       | リソースの取得                                           |
| POST      | 子リソースの作成、リソースへのデータの追加、その他の処理 |
| PUT       | リソースの全体的な更新、リソースの作成                   |
| PATCH     | リソースの部分的な更新                                   |
| DELETE    | リソースの削除                                           |
| HEAD      | リソースのヘッダーの取得                                 |
| OPTIONS   | リソースがサポートしているメソッドの取得                 |
| CONNECT   | プロキシーへのトンネリング要求                           |
| TRACE     | 経路の調査                                               |

この表で説明しきれないことを解説していきます。

## POSTとPUTの作成処理の違い

**作成するリソースをクライアントが明示できるかどうか**が異なります。

POSTは作成するリソースをクライアントが明示できません。  
たとえば、リクエストURIが`/articles`のようになります。

一方で、PUTは作成するリソースをクライアントが明示できます。  
たとえば、リクエストURIが`/articles/1`のようになります。

## PUTとPATCHの更新処理の違い

**差分アップデートができるかどうか**が異なります。

PUTはリソースの情報全てを明示しないといけません。  
たとえば、名前とメアドをもつユーザーリソースを考えます。  
名前だけを変更したい場合でも、リクエストボディに名前とメアド全てを記述する必要があります。

一方で、PUTは差分だけでOKです。  
たとえば、先ほどのユーザーリソースを名前だけ変更したい場合、リクエストボディに名前だけを記述して更新できます。

## 冪等性と安全性

HTTPメソッドにおいて着目される性質があります。  
それが冪等性と安全性です。

冪等とは、**ある操作を何回行っても結果が同じこと**を意味します。  
たとえば、DELETEは1回実行しようが100回実行しようが同じ結果になるので冪等です。  
一方で、POSTは実行回数分子リソースが作成されうるので冪等ではありません。

安全とは、**操作対象のリソースの状態を変化させないこと**を意味します。  
たとえば、GETはリソースを取得するだけなので安全です。  
一方で、DELETEはリソースを削除するので安全ではありません。

各メソッドに対する冪等性と安全性の有無は以下のとおりです。

| メソッド | 冪等性 | 安全性 |
| :------- | :----- | :----- |
| GET      | ◯     | ◯     |
| POST     | ×     | ×     |
| PUT      | ◯     | ×     |
| PATCH    | ×     | ×     |
| DELETE   | ◯     | ×     |
| HEAD     | ◯     | ◯     |
| OPTIONS  | ◯     | ◯     |
| CONNECT  | ×     | ×     |
| TRACE    | ◯     | ◯     |
